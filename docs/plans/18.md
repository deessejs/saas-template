## TL;DR

`/api/ready` returns `200 { status: "ready" }` unconditionally. After this PR, it pings Postgres and returns `503 { status: "not ready" }` when the DB is unreachable.

---

## Issue summary

The `/api/ready` endpoint wraps `c.json()` in a try/catch, but `c.json()` never throws â€” it constructs a Response object. The endpoint always returns `200 { status: "ready" }`, providing a false signal to Kubernetes and load balancers. The fix is documented in `docs/guides/better-auth/setup.md Â§189 and involves calling `db.execute(sql`SELECT 1`)` inside the try block.

## Step-by-step implementation

### Step 1 â€” Add `db` import to packages/api/src/index.ts

`packages/api/src/index.ts`

Add two imports:
- `db` from `@workspace/database` (already exports `db` from `./client.js`)
- `sql` from `drizzle-orm/postgres-js` (transitive dep, already available)

The `sql` tag template literal is the correct drizzle way to write raw SQL that returns a query result, as opposed to `sql.raw()` which returns raw bytes.

The `db` singleton is serverless-safe (pool init on first access). It is safe to call in the readiness handler since Hono creates one handler instance per request.

### Step 1.1 â€” Verify `sql` tag availability

`drizzle-orm/postgres-js` re-exports `sql` from `postgres`. The `execute(sql`SELECT 1`)` call is the standard drizzle-pattern for a liveness check. If `db` is a passthrough object (no-op during `auth:generate`), the execute will no-op gracefully â€” `/ready` returns `200` when `DATABASE_URL` is absent.

### Step 2 â€” Replace dead try/catch with real DB ping

`packages/api/src/index.ts`

Replace the current broken handler:

```ts
// before (dead code)
api.get("/ready", async (c) => {
  try {
    return c.json({ status: "ready" })   // c.json() never throws
  } catch {
    return c.json({ status: "not ready" }, 503)
  }
})

// after (correct)
api.get("/ready", async (c) => {
  try {
    await db.execute(sql`SELECT 1`)
    return c.json({ status: "ready" })
  } catch {
    return c.json({ status: "not ready" }, 503)
  }
})
```

`db.execute(sql`SELECT 1`)` throws on connection failure (ECONNREFUSED, ETIMEDOUT, pool exhausted) â€” exactly when we want to return 503. When Postgres is healthy, it returns the query result and we return 200.

## Risks / edge cases

- **`db` passthrough when `DATABASE_URL` absent**: During `auth:generate` or `pnpm db:generate` (CLI context), `db` is a no-op passthrough object. `db.execute(sql`SELECT 1`)` on the passthrough may silently succeed or no-op. This means `/ready` returns `200` even without a DB â€” acceptable for local dev where the DB may not be running. Production always has `DATABASE_URL` set.
- **Connection timeout**: `db.execute` has a default timeout. Slow-starting Postgres will return 503 briefly then transition to 200 once the pool is healthy â€” this is the correct behavior.
- **No migration needed**: Zero schema or data migration. Pure code fix.
